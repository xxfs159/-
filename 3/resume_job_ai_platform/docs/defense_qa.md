# 答辩问答准备

## Q1：你们的项目为什么叫 RAG？

RAG 是“检索增强生成”。本项目中，岗位 JD 数据库就是外部知识库。系统先根据用户简历检索最相关岗位，再基于检索结果生成简历优化建议、求职信和面试要点。因此符合 RAG 的基本思想。

## Q2：为什么不用真实招聘网站数据？

真实招聘网站存在登录、反爬和数据合规问题，课堂展示时不稳定。为了保证复现性，我们先用 CSV 文件模拟岗位知识库。后续可以扩展爬虫或开放 API 自动更新。

## Q3：为什么默认使用 TF-IDF，而不是 BGE embedding？

BGE embedding 效果更好，但需要下载模型和占用更多算力。项目默认使用 TF-IDF 是为了保证普通电脑可以直接运行。系统结构已经模块化，后续只需要替换向量化函数即可接入 BGE。

## Q4：匹配度是怎么计算的？

综合匹配分由三部分组成：语义相似度、技能重合度和方向匹配度。公式为：

综合匹配分 = 0.55 × 语义相似度 + 0.35 × 技能重合度 + 0.10 × 方向匹配度。

这样既考虑整体文本相似性，也考虑岗位硬技能要求和用户求职偏好。

## Q5：系统生成的求职信会不会编造经历？

默认模板不会编造不存在的经历，只会基于已识别技能和岗位要求生成保守表述。如果接入 DeepSeek，也在 Prompt 中明确要求“不编造经历”。

## Q6：项目的创新点是什么？

主要有三点：

1. 不只是岗位推荐，还覆盖简历优化、求职信和面试准备；
2. 推荐结果可解释，能显示匹配技能和缺失技能；
3. 系统轻量可复现，同时保留接入 DeepSeek、BGE、向量数据库的扩展空间。

## Q7：如何证明技术有效性？

可以通过示例简历演示：当选择不同求职方向时，Top-K 推荐岗位会发生变化；当简历中增加某些技能时，对应岗位的技能匹配度会提升。系统还会输出可解释的匹配原因，便于人工检查。

## Q8：如果岗位知识库很大，系统如何扩展？

可以把岗位 JD 向量提前计算并存入 FAISS、Chroma 或 Milvus 等向量数据库。用户上传简历后，只需要计算一次简历向量，再进行 Top-K 向量检索。

## Q9：项目如何接入 DeepSeek？

在环境变量中配置 `DEEPSEEK_API_KEY`。系统的 `generator.py` 已经预留 DeepSeek API 调用函数。如果没有 Key，则自动使用本地模板，保证演示不受影响。

## Q10：项目还有哪些改进方向？

- 使用 BGE embedding 提高语义匹配效果；
- 接入真实岗位爬虫；
- 使用 LLM 进行更精细的简历结构化解析；
- 增加用户反馈机制；
- 支持多版本简历对比；
- 增加岗位技能图谱和学习路径推荐。
